# Option 3 Investigation: Seamless Terminal Tab Return

_Last updated: 2026-04-08_

Status note: Termsy now uses the local `Packages/TermsyGhostty` package rather than the old remote `libghostty-spm` / `GhosttyTerminal` dependency. References below to the old wrapper path are historical and are only relevant where they describe the same underlying surface-centric API limits.

See also: [`docs/vt-custom-renderer-architecture.md`](vt-custom-renderer-architecture.md) for a concrete architecture built on the public `libghostty-vt` API instead of `ghostty_surface_t`.

## Executive summary

This document investigates a **true option 3** for Termsy: making leaving and returning to a terminal tab feel as seamless as possible, while treating **tab switching** and **app background/foreground** as one coherent design.

### Short version

- The **current junk injection bug** is caused by **Termsy replaying raw terminal output into a fresh Ghostty surface** after the old one was destroyed.
- For **tab switching inside a running app**, the smallest architecture that truly fixes this is primarily a **Termsy lifecycle change**, not a Ghostty-core feature project:
  - keep one live terminal surface per tab
  - stop destroying it on deselection
  - stop using transcript replay as the restoration mechanism
- For a stricter, more general version of option 3 where a terminal can survive **without a bound `UIView` at all**, the blocker moves below Termsy:
  - the current Swift bridge layer does not expose detach/reattach or a separate terminal-core abstraction
  - the public `libghostty` API also does not expose detach/reattach, headless preservation, or snapshot/restore of full terminal state
- For **real iOS background suspension**, no purely local design can guarantee continuous SSH/terminal progress once the process is suspended. The realistic continuity boundary is:
  - **while the app remains resident**: preserve local live surfaces
  - **after suspension / disconnect**: recover via reconnect + remote session persistence (`tmux`)

### Recommendation

For **Termsy as an iOS app**, the recommended option 3 is:

1. **App-level live tab residency** as the main architecture
   - each open tab owns a persistent live Ghostty surface while the app is running
   - hidden tabs are hidden/occluded, not destroyed
   - incoming SSH output continues to advance the same terminal state
2. **Scene/app lifecycle integration**
   - on app inactive/background: unfocus/occlude surfaces instead of tearing them down
   - on foreground: reactivate surfaces if connection survived
   - if suspension killed the connection, fall back to reconnect + `tmux`
3. Treat **detach/reattach or terminal-core/render split** as a longer-term Ghostty/libghostty evolution, not the first move required to solve the user-facing problem in Termsy

---

## Goals

A successful option 3 for Termsy should make leaving and returning feel seamless across:

- switching to another tab and back
- brief app inactive / focus loss
- app background / foreground under practical iOS constraints

In this document, “seamless” means:

- no raw escape-sequence junk appears at the shell prompt on return
- the tab returns showing the **current** terminal state, not a replay artifact
- hidden tabs continue progressing while the app is still alive
- the design states clearly what happens once iOS suspends the app

## Current architecture

### Relevant files

- `Termsy/Views/ContentView.swift`
- `Termsy/Views/ViewCoordinator.swift`
- `Termsy/Views/TerminalTabBarController.swift`
- `Termsy/Views/TerminalView.swift`
- `Termsy/SSH/SSHTerminalSession.swift`
- `Termsy/TermsyApp.swift`

### How tab switching works today

Only the selected tab is rendered in `ContentView`:

- `TerminalContainer` renders `TerminalHostRepresentable(tab: tab).id(tab.id)` for the selected tab only.

When selection changes in `ViewCoordinator.selectTab(_:)`:

- previous tab gets `sshSession.enterBackground()`
- new tab gets `sshSession.enterForeground()`

When the selected tab view is torn down:

- `TerminalHostRepresentable.dismantleUIViewController` calls `teardownTerminal()`
- `teardownTerminal()` calls `terminalView?.stop()`
- `TerminalView.stop()` calls `ghostty_surface_free(...)`

So the Ghostty surface is destroyed whenever that tab is no longer the selected rendered tab.

### How state restoration works today

`SSHTerminalSession` currently maintains a **transcript file**:

- all inbound SSH data is appended to disk
- if a new surface is created later, `replayIfNeeded()` replays the transcript into the new `TerminalView`

That means the restoration model is:

```text
SSH bytes -> transcript file -> fresh Ghostty surface -> replay historical bytes
```

### How terminal-generated bytes are handled

`TerminalView` is configured with host-managed I/O.

Its `onWrite` callback is documented as receiving terminal-produced bytes including:

- user input
- query responses

`TerminalHostController.setupTerminal()` wires those bytes directly to SSH:

```text
Ghostty terminal output -> SSHConnection.send(...)
```

## Root cause of the junk injection bug

The junk like:

- `62;22;52c`
- `>|ghostty 1.3.1`

is consistent with terminal capability / identification replies.

The failure mode is:

1. a tab is hidden
2. its Ghostty surface is destroyed
3. historical terminal output is later replayed into a fresh surface
4. some historical bytes are not just “screen paint”; they include VT queries
5. Ghostty interprets those queries as live input to the emulator and emits replies
6. those replies are forwarded upstream over SSH as fresh terminal output/input
7. if nothing on the remote end is waiting for them anymore, the shell simply echoes them as visible junk

So the bug is not random corruption. It is a direct consequence of using **raw VT replay** as a state restoration mechanism.

## Layer-by-layer blocker attribution

## 1. Termsy layer

### What Termsy is doing that causes the user-visible problem

The immediate bug is primarily caused by Termsy’s lifecycle choices:

- only one tab’s terminal surface is kept alive at a time
- deselected tabs destroy their Ghostty surface
- restoration is byte replay rather than live-state preservation

### What Termsy can already do with the existing API

With the current embedded `libghostty` C API exposed through `TermsyGhosttyKit`, Termsy can already:

- create one live surface per tab
- keep it alive as long as its backing `UIView` stays alive
- stop or reduce active rendering by controlling focus and display activity
- avoid replay entirely for in-process tab switching

### Termsy-level conclusion

For **tab switching while the app is alive**, the main blocker is **Termsy’s current ownership/lifecycle model**, not an inherent Ghostty limitation.

---

## 2. Swift bridge layer

### Current bridge capabilities

Termsy now vendors `libghostty` through `Packages/TermsyGhostty`. The same surface-centric limits that mattered during the old remote-wrapper investigation still apply to the current bridge layer.

From the public API and bridge code, the current integration already exposes or can reach:

- `ghostty_surface_new(...)`
- `ghostty_surface_free(...)`
- `ghostty_surface_write_buffer(...)`
- `ghostty_surface_process_exit(...)`
- `ghostty_surface_set_focus(...)`
- `ghostty_surface_set_occlusion(...)`
- `ghostty_surface_set_size(...)`

The current UI bridge layers also assume a surface is created when attached to a platform view and freed when detached.

### What the bridge layer does **not** currently provide

It does not provide an app-facing API for:

- detaching a live terminal from one `UIView` and reattaching it to another
- keeping a terminal core alive with **no view bound at all**
- snapshotting/restoring full terminal state
- separating terminal state ownership from surface ownership

### Bridge-level conclusion

The bridge layer is not the reason Termsy cannot keep tabs seamless **today** if it keeps surfaces alive.

But the bridge layer **is** a blocker if Termsy wants a more advanced design where a live terminal survives independently of a `UIView`.

---

## 3. Core `libghostty` / Ghostty layer

### Current public C API shape

The public embedding API is surface-centric:

- a `ghostty_surface_t` is created with platform config including a `UIView`
- the surface is then used for input, sizing, drawing, focus, occlusion, etc.
- destroying the surface frees the object that owns terminal/renderer runtime state

### What is missing in the public C API

The public API does **not** expose:

- surface detach / reattach
- headless terminal preservation
- a separate `terminal session` / `terminal core` object independent of a surface
- full snapshot/export + restore/import of terminal state

### Architectural implication

The deeper Ghostty model is still centered around the **surface** as the owner of:

- terminal state
- renderer thread
- I/O thread
- runtime platform binding

That means a strict interpretation of option 3—“keep the live terminal core running without any bound UI object”—is currently below what the public API exposes.

### Core-layer conclusion

For **general detached/headless preservation**, the blocker is not just the wrapper. It reaches into the public `libghostty` API and likely the underlying Ghostty surface architecture.

## Capability matrix

| Capability | Termsy today | Wrapper today | Core public API today | Notes |
| --- | --- | --- | --- | --- |
| Keep a selected tab live | Yes | Yes | Yes | Already done for the visible tab |
| Keep multiple tabs live at once | Not currently, but app can be refactored to do it | Yes | Yes | Requires different ownership/UI strategy |
| Stop drawing without freeing terminal state | Partially | Partially | Yes | Focus/occlusion APIs exist |
| Destroy and later recreate surface from raw VT replay | Yes | Yes | Yes | This is the current bug-prone approach |

| Keep terminal core alive with no bound view at all | No | No | No public API | Would require deeper architectural change |
| Serialize full terminal state and restore later | No | No | No public API | Text extraction APIs are not enough |

## Why pure snapshot/restore is not enough for Termsy

This is the key design point.

Termsy’s tabs are backed by live SSH connections. While a tab is hidden, the remote side may continue sending:

- prompt redraws
- TUI updates
- cursor/color/device queries
- alternate-screen changes
- mouse/cursor mode changes

If no live terminal core is consuming those bytes while hidden, then a future restore still needs to handle the missing interval somehow.

That leaves only two choices:

1. **Replay raw VT later**
   - this is the current class of bug
2. **Keep a live terminal core consuming bytes while hidden**
   - this is the real fix

Therefore, for Termsy, a true option 3 is fundamentally about **continuous hidden-state advancement**, not a one-time snapshot.

## Candidate architectures

## A. App-only live surface per tab

### Description

Each `TerminalTab` owns a persistent terminal host + `TerminalView` + Ghostty surface for as long as the tab is open.

When a tab becomes hidden:

- do **not** destroy its `TerminalView`
- do **not** free its Ghostty surface
- mark it unfocused and occluded
- stop or reduce active rendering work
- keep feeding SSH output into the same live surface

When the user returns:

- show the same existing view/surface again
- refocus it
- resume drawing
- no transcript replay needed

### What changes in Termsy

#### Ownership model

Today:

- `ContentView` only renders the selected tab
- unselected tab hosts are dismantled

Recommended:

- keep a persistent host per open tab
- `TerminalContainer` renders all tab hosts in a `ZStack`
- only one is visible and interactive at a time
- hidden hosts remain in memory

#### Tab state transitions

On deselect:

- `terminalView.setDisplayActive(false)`
- add a `setOccluded(true)` wrapper around `ghostty_surface_set_occlusion(...)`
- resign first responder
- keep `sshSession.terminalView` attached

On reselect:

- `setOccluded(false)`
- `setDisplayActive(true)`
- become first responder
- no transcript restoration step

#### SSH session behavior

- remove transcript-based restoration as the steady-state design
- live SSH bytes continue feeding the same surface even while hidden

### Strengths

- smallest architecture that truly fixes the current bug in-process
- no new third-party dependency
- likely no Ghostty-core changes required
- preserves real terminal state instead of replaying bytes

### Weaknesses

- more memory use per open tab
- hidden tabs still keep live Ghostty surfaces and associated state
- view lifetime and tab lifetime become tightly coupled
- this is still not a general “terminal detached from UI” solution

### Feasibility verdict

**High.** This is the best near-term option for Termsy.

---

## B. Detach/reattach a live terminal to different `UIView`s

### Description

A live terminal would survive even when no concrete `UIView` is bound. Later, a new `UIView` could attach to the same terminal.

### Why this is attractive

It gives most of the seamlessness of option 3 without forcing a live hidden `UIView` hierarchy for every tab.

### What is missing today

No public API currently allows:

- detaching a surface from its original platform view
- reattaching it to a new platform view
- keeping the terminal alive without the view-bound surface object

### Practical conclusion

This cannot be achieved by Termsy alone. It would require at least:

- new bridge API
- and most likely new `libghostty`/Ghostty embedding support

### Feasibility verdict

**Medium-to-low as a first move.** Architecturally appealing, but not the smallest real solution for Termsy.

---

## C. Split terminal core from render surface

### Description

Introduce a first-class terminal core object that owns:

- terminal state
- VT parser
- I/O integration
- modes / scrollback / screen buffers

Then render surfaces become attachable views over that core.

### Why this is the cleanest general design

This is the most principled option because it matches the real conceptual split between:

- terminal emulation state
- rendering target / UI lifetime

### What it would require

Likely upstream changes in Ghostty/libghostty such as:

- a new terminal/session handle separate from `ghostty_surface_t`
- surface creation from an existing terminal core
- attach/detach semantics for render targets
- clarified renderer-thread behavior when no UI is attached

### Feasibility verdict

**Best long-term architecture, highest cost.** Not the first implementation Termsy should rely on.

---

## App background/foreground under practical iOS constraints

This is where the architecture must be honest.

## While the app is still resident

If the app merely becomes inactive or briefly backgrounded but remains alive:

- the app can keep live surfaces in memory
- the selected tab and hidden tabs can continue to hold their terminal state
- on foreground, the app can refocus and redraw without replay

This fits option A.

## After real iOS suspension

Once the process is suspended:

- continuous local SSH progress cannot be assumed
- continuous terminal-core advancement cannot be assumed
- local seamlessness ends at the process boundary

At that point the only realistic continuity source is remote state, i.e.:

- reconnect the SSH session
- reattach to `tmux` if configured

## Therefore, true option 3 for Termsy should mean

### Inside one running app lifetime

- local live terminal continuity
- no destructive tab switching
- no transcript replay restoration

### Across iOS suspension

- best effort local preservation if the process survives briefly
- otherwise reconnect + remote session continuity

This is the correct iOS-specific definition of “true option 3” for Termsy.

## Recommended architecture

## Recommendation: implement option A now, document B/C as future engine work

### Why

It is the smallest architecture that:

- actually fixes the current bug
- gives real seamless tab return inside the running app
- aligns with practical iOS lifecycle constraints
- does not require inventing unsupported detach/headless APIs first

### Recommended design statement

> For Termsy, option 3 should mean that every open tab retains a live terminal surface for the duration of the app’s active lifetime, and hidden tabs become non-visible/non-focused rather than destroyed. App background/foreground should preserve those live tabs when the process remains resident, and fall back to reconnect + `tmux` continuity when iOS suspension breaks local continuity.

## Proposed Termsy object model

## `TerminalTab`

Owns long-lived tab state:

- `Session`
- `SSHTerminalSession`
- persistent terminal host/controller identity
- visibility/focus state
- reconnect policy state

## `TerminalHostController`

Becomes a long-lived per-tab host instead of an ephemeral representable product.

Responsibilities:

- own one `TerminalView`
- keep it alive until tab close
- switch between visible / hidden / inactive modes
- manage first-responder and display activity

## `TerminalView`

Remains the bound Ghostty surface owner for the app-only design.

Needs two explicit states:

- **display active**: focused, display link running
- **occluded**: hidden, not actively rendering, but terminal state still alive

Potential additions:

- `setOccluded(_:)` -> `ghostty_surface_set_occlusion(...)`
- clearer separation between “stop drawing” and “free surface”

## `SSHTerminalSession`

Should no longer treat transcript replay as the normal restoration path for tab switching.

Long-term direction:

- live bytes go straight into the persistent tab surface
- transcript/replay can be removed entirely or retained only as an explicit fallback/debug path

---

## Scene/app lifecycle design

Termsy currently relies mostly on `UIApplication.didBecomeActiveNotification` to reconnect if the transport died. That should be expanded into an explicit lifecycle policy.

## On tab deselect

- tab surface stays alive
- focus false
- occluded true
- display link paused

## On tab select

- occluded false
- display active true
- first responder restored

## On app will resign active / enter background

For all open tabs:

- set focus false
- set occluded true
- stop display links
- keep surfaces alive while the process remains resident

## On app did become active

For all open tabs:

- if the SSH transport is still alive, reactivate selected tab display/focus
- hidden tabs remain alive but occluded
- if selected tab transport died, reconnect and rely on remote state (`tmux`) if available
- optionally reconnect hidden tabs lazily on selection rather than all at once

---

## Implementation phases

## Phase 1: stop destructive tab switching

Change Termsy so that:

- open tabs keep persistent hosts/views
- `ContentView` no longer causes non-selected tabs to dismantle their terminal hosts
- tab switching becomes show/hide, not destroy/recreate

Expected result:

- the current junk injection bug should disappear because there is no replay into a fresh surface

## Phase 2: add explicit hidden-surface behavior

Add or expose:

- `setOccluded(true/false)`
- clear display activity transitions
- app lifecycle hooks for background/inactive transitions

Expected result:

- hidden tabs consume little rendering work but preserve state

## Phase 3: simplify `SSHTerminalSession`

Remove transcript/replay from the normal path.

Expected result:

- SSH session model becomes straightforward again: live bytes feed live terminal state

## Phase 4: strengthen reconnect policy across suspension

Define a clear fallback policy:

- if process survives: seamless local continuity
- if transport dies: reconnect
- if `tmux` configured: remote continuity restored
- if no `tmux`: fresh shell is expected behavior

## Phase 5: optional future Ghostty exploration

If Termsy later needs lower memory usage or UI-detached preservation, investigate:

- detach/reattach API
- terminal-core/render split

---

## Risks and tradeoffs

## Memory / resource cost

Keeping one live surface per tab increases memory compared with the current destroy/rebuild model.

Mitigations:

- pause display links for hidden tabs
- mark hidden tabs occluded
- consider upper bounds or lazy policies if many tabs are open

## App lifecycle complexity

The app must now clearly separate:

- hidden tab
- inactive app
- backgrounded but resident app
- resumed after suspension with dead transport

This is added state-management complexity, but it is conceptually cleaner than transcript replay.

## Remote continuity still depends on remote tooling

Across real iOS suspension, full continuity still depends on remote persistence such as `tmux`.

That is not a design flaw in Termsy; it is the realistic boundary for an SSH terminal app on iOS.

---

## Final conclusion

### Where the blocker lives

- The **current visible bug** is primarily a **Termsy architecture problem**.
- The blocker for a stricter **UI-independent live terminal** lives in **bridge + core Ghostty/libghostty API design**.

### What Termsy should do next

Termsy should not pursue transcript replay as its restoration model.

Instead, it should adopt:

- **persistent live per-tab terminal surfaces** while the app is running
- **focus/occlusion/display lifecycle management** instead of destroy/recreate
- **reconnect + `tmux` fallback** across actual iOS suspension

That is the smallest real option 3 that matches Termsy’s goals and the platform’s constraints.
